# orvk の思想

この文書は、orvk の設計判断そのものではなく、設計判断の背後にある思想・文脈をまとめた読み物である。
個々の意思決定は [docs/design/](design/README.md) の番号付き design doc に記録されており、本文書はそこから参照される前提を提供する。

## Vulkan 専用ラッパーという文脈

orvk は複数の GPU API を抽象化する RHI ではなく、Vulkan 専用の単体ラッパーライブラリである。
対象を Vulkan、さらに `VK_EXT_descriptor_heap` が使える環境に絞り、fallback backend を持たない。

この絞り込みは制約ではなく存在理由である。
複数 API の最大公約数に合わせた抽象は、descriptor set のバインド操作、リソース状態遷移の手動管理、API ごとの同期モデルの差異を利用者へ漏らすか、最も貧しい API に合わせて表現力を削るかの二択を迫る。
Vulkan ひとつ、descriptor heap 前提に絞れば、記述子バインドという概念そのものを利用者から消し、リソースは世代付き handle とレジストリで一貫して管理し、同期はタスクグラフの access 宣言から一意に導出できる。
広く浅い互換性より、対象領域での深い一貫性を選ぶ。

一方で、ラッパーの語彙に無い Vulkan 機能を封じることはしない。
raw handle への unsafe な escape hatch を明示的な境界として提供する。
安全な語彙の内側と、利用者が責任を負う unsafe の外側を、型と API の境界としてはっきり分ける。

## easy ではなく simple を選ぶ

orvk の設計では、短く呼び出せること、初見で少ないコード量に見えること、既存の慣れた API に近いことを、概念分離より優先しない。

ここでいう **simple** は「簡単そうに見える」ことではない。
概念が絡み合っておらず、責務が分離され、境界が一方向に読める状態を指す。
**easy** は、利用者や実装者にとって手近で、慣れていて、すぐ呼べるという主観的な近さを指す。
この区別は Rich Hickey の講演 "Simple Made Easy"(Strange Loop 2011)に基づく。
Hickey は simple を構造的に絡んでいない状態、easy を人間にとって近い状態として切り分け、両者を混同すると短期的に使いやすい構造が長期的な複雑性を作ると論じた。

orvk でこの原則が効く場所は多い。
リソースの handle 管理と GPU hazard(同期)管理を一つの API に混ぜない。
リソース生成・コマンド記録・submit・完了観測を一つの型に押し込まず、責務ごとの境界として明示する。
安全な記録語彙と raw Vulkan への escape hatch を同じ顔の API に混ぜず、unsafe の境界で分ける。
利用側コードを短くする helper は、概念境界を保ったうえで後から足す。
easy は simple な構造の上に置くものであり、simple な構造を壊して easy を作らない。

この選択は、orvk の利用者がレンダラーや GPU アプリケーションを書く専門層であることと表裏一体である。
最小サンプルの行数を競うライブラリではなく、長く保守されるレンダラーの基盤として、構造の読みやすさと正しさの保証を優先する。

## 同期は宣言から導出する — 状態の経路を一本化する

Vulkan の同期(バリア・semaphore・image layout 遷移)は、手で書くと同じ判定が複数箇所に分散し、一箇所の変更が他の箇所と歩調を崩して drift する。
orvk はこれを、タスクグラフへの access 宣言という一本の経路に集約する。
描画・計算はタスクとして記録し、各タスクがどのリソースにどうアクセスするかを宣言し、必要な同期はそこから導出する。

経路を一本にすると、入口を増やせる。
単一プロセス内の複数サブシステム(たとえばアプリの複数のレンダリング系)が 1 つの Device とリソースレジストリを共有しても、同期の判断点は分散しない。
また window ありの present も、window なしのヘッドレスな submit → readback も、同じ記録と同期導出の上に立つ。
headless 実行が「後付けの mock」ではなく「もう一つの入口」になるのは、この一本化の帰結である。

## 正しさを後から補わない

orvk の設計を貫くもう一つの態度は、「曖昧なまま進めて後から補う」ことを基盤仕様について拒否することである。
handle の同一性と世代管理、リソースの lifetime と破棄の遅延規則、submit 失敗時の状態、完了観測の意味論 — これらは最初から仕様として定義し、test 可能な invariant として扱う。

同時に、すべてを最初に固定することも拒否する。
crate 分割のような構造の物理的な形は境界が実装で安定するまで遅延し、最適化は必要が証明されるまで実装しない。
固定すべきもの(正しさの仕様)と遅延すべきもの(構造の物理的形、最適化)を区別すること自体が、simple over easy の実践である。

## まとめ

- 対象は Vulkan 専用。`VK_EXT_descriptor_heap` 前提に絞り、fallback を持たず、深い一貫性を取る。raw への escape hatch は unsafe の境界として明示する。
- simple > easy。概念境界を壊す近道は作らず、helper は境界の上に後から足す。
- 同期は access 宣言からの導出に一本化し、Device 共有も headless 実行も同じ入口に立たせる。
- 正しさの仕様は最初から定義し、構造の固定と最適化は遅延する。
