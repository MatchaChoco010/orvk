# Markdown 執筆の作法

このドキュメントは **markdown を書く・編集するとき** に読む。
design doc・`docs/harness/`・README・skill の `SKILL.md` など、このリポジトリのすべての markdown に適用される共通ルールである。

## 文の途中で、見た目のために改行しない

行の折り返し(見た目の横幅を整える目的)だけのために、**文の途中で改行を入れてはいけない**。
長い文を見た目のために複数行へ割らず、折り返しはエディタのソフトラップ(word wrap)に任せる。

文の途中で見た目のために改行すると、word wrap のあるエディタで開いたときにエディタ自身の折り返しと手で入れた改行が混ざり、中途半端な改行だらけになって逆に非常に読みにくくなる。
差分も、文の途中に 1 文字挿入しただけで以降の折り返し位置がずれて行全体が書き換わり、レビューしづらくなる。

## 改行は「意味の区切り」で行う

改行してよいのは意味の区切りであって、見た目の都合ではない。
次は意味の区切りなので改行してよい(むしろすべき)。

- 段落の区切り(空行)。
- 文末。日本語の散文では **一文ごとに改行する「一文一行」を標準とする**(→ `japanese-tech-writing` skill)。文末(。)での改行は意味の区切りなので本ルールに反さず、差分も読みやすくなる。
- リスト項目の区切り(項目ごとに 1 行。1 項目の中の文は途中で割らない)。
- コードブロック・コードフェンス内の改行(コードの一部)。
- 表(table)の行。
- 見出し・引用ブロックなどの行構造。

設定的・表形式など散文でない短い記述は、一文一行にこだわらず 1 段落 1 行で書いてもよい。
いずれにせよ核は「文の途中を見た目のために割らない」ことである。

## 例

悪い例(1 つの文を、見た目の横幅のために途中で折り返している):

```md
このルールは設計ドキュメントだけでなく、リポジトリのすべての
markdown に適用される。文の途中で改行すると、word wrap の
あるエディタで読みにくくなる。
```

良い例(一文一行。文末で改行し、文の途中では割らない):

```md
このルールは設計ドキュメントだけでなく、リポジトリのすべての markdown に適用される。
文の途中で改行すると、word wrap のあるエディタで読みにくくなる。
```

## 技術用語は慣例に従う(一般語を無断で術語化しない)

技術概念の名前には、その分野で慣例的に使われる用語(カタカナ語・英語をそのまま含む)だけを使う。
一般的な日本語の語彙(例: 台帳、正典)に、ことわりなく特別な技術的意味を持たせて概念名として使わない。
一般語の術語化は未定義語になり、読者の理解を妨げる不自然な日本語を生む。
複数のドキュメントをまとめて指す名前が必要な場合も、名前を発明せずに対象を直接列挙して参照する(名付けがどうしても必要なら、個別のドキュメントや skill の内部ではなく CLAUDE.md のようなプロジェクトレベルの場所で定義する)。
