# ハーネス編集の作法

このドキュメントは **ハーネスを編集するときに読む**。
ハーネスとは、エージェント(Claude Code)向けの恒久的な指示・規約・道具立ての総称で、具体的には次を指す。

- `CLAUDE.md`(常時読み込まれる standing 規約)
- `.claude/skills/` 配下の skill
- `docs/harness/` 配下の参照ドキュメント(本ファイルを含む)
- `docs/design/README.md` / `template.md`(design doc のルール)
- `scripts/` の再利用スクリプト

ハーネスの変更は、プロダクトコードと同じく **Issue + feature ブランチ + ゲーティング PR** で行い、マージはユーザーが行う(詳細は [git-and-pr.md](git-and-pr.md))。
本ファイルはその「中身をどう書くか」を定める。

## 1. 既存を読み、マージ・整理してから書く(継ぎ足し禁止)

ユーザーの指摘やフィードバックを反映するとき、**該当箇所の末尾に新しい段落を継ぎ足すだけ**にしない。
継ぎ足しを繰り返すと、ハーネスは整理されないまま冗長に伸び、同じ話題が複数箇所に分散し、読み手(=次にそれを使うエージェント)が一意に読み取れなくなる。

加筆・修正の前に必ず次を行う。

1. **既存のハーネス全体の関連箇所を読む。** 加筆しようとしている内容が、既にどこかで(CLAUDE.md・別の skill・別の docs/harness)語られていないかを確認する。
2. **重複・近接する記述はマージする。** 同じ規則を2か所に置かない。既存の記述に統合できるなら、新しい節を足すのではなく既存の節を書き直して取り込む。
3. **整理してから書く。** 追記後の文書が、最初から一貫して書かれたかのように読める状態にする。「あとから足したパッチの層」に見える状態にしない。節の順序・見出し・粒度を必要なら組み替える。
4. **一意に読めるか確認する。** 誤読の余地がある表現、複数の解釈ができる指示を残さない。skill は再現性が命なので、読み手によって挙動が変わる書き方をしない。

目安: 「この変更で文書は短く・明確になったか、それとも長く・散らかったか」を自問する。
後者なら整理し直す。

## 2. CLAUDE.md の整理の規律 — 常時必要な規約だけを置く

`CLAUDE.md` は **毎セッション常に読み込まれる**。
だから CLAUDE.md には、**事実上どの作業でも関係する standing な規約(ゲート)だけ**を置く。
特定の作業のときにしか要らないルールを CLAUDE.md に書かない。

加筆先を決める判断基準:

- **常時必要(ほぼ毎タスク関わる)** → CLAUDE.md に、既存と**マージして**置く。例: Pre-v1 API 方針、誰が何をマージするか、PR はレビュー単位、ハーネスも Issue+PR。
- **特定作業のときだけ必要(条件付き)** → **on-demand ドキュメント(`docs/harness/`)または skill** に置き、CLAUDE.md からは**短いポインタ1行**で参照する。例: scripts の書き方(→ [scripts.md](scripts.md))、Issue/PR の詳細手順(→ [git-and-pr.md](git-and-pr.md) / `/pr-workflow` skill)、ハーネス編集の作法(本ファイル)。

「念のため CLAUDE.md に書いておく」をしない。
条件付きルールを CLAUDE.md に積むと、毎セッションのコンテキストを食い、本当に常時必要な規約が埋もれる。
迷ったら「この規則は、これと無関係な作業をしているエージェントにも読ませる価値があるか」を問う。
無ければ on-demand 側に置く。

参照ドキュメントを新設したら、CLAUDE.md(または親の skill/doc)にポインタを張り、[README.md](README.md) の索引に「いつ読むか」とともに登録する。
ポインタは「いつ読むか」が分かる1行にする(例:「scripts を書くときは [scripts.md] を読む」)。
ただしポインタを張るだけで終わらせず、**読まれる仕組み**まで用意する(§5)。

## 3. skill も同じ — 1つの skill に無関係な条件を積まない

skill にも同じ原則が効く。
ユーザーの指摘が来るたびに1つの skill の末尾へ無関係な条件を継ぎ足さない。
skill は単一の役割に閉じ、その役割に必要なことだけを、整理された形で持つ。
別の関心事が育ってきたら、別の skill か参照ドキュメントに分ける。
skill 間で重複する手順は一方を正にしてもう一方からポインタで参照する(例: `/pr-workflow` は方針の正を [git-and-pr.md](git-and-pr.md) に置き、自身はコマンド手順に徹する)。

## 4. skill 本文から repo 内の文書を指すポインタ

skill の `SKILL.md` 本文は、skill 起動時にそのまま**コンテキストへ展開される**(ファイルとして開かれるのではない)。
本文中の markdown リンクは**自動で解決・ロードされない** — リンク先の中身が要るときはエージェントが `Read` で開く。
したがって skill から repo 内の文書を指すときは次のようにする。

- **パスはリポジトリルート起点**で書く(例: `docs/harness/git-and-pr.md`)。エージェントの cwd はリポジトリルートなので、この形なら `Read` で確実に開ける。`../../../docs/...` のような **SKILL.md 位置を基点とする相対パスは使わない**(基点が曖昧で脆い)。リンクにするなら表示テキストも href もルート起点に揃えるか、いっそインラインコードのパスで書く。
- **CLAUDE.md は常時コンテキストにある。** CLAUDE.md から既に辿れる文書を、skill 側でわざわざ deep-link して重複させない。「方針の正は CLAUDE.md『…』と、そこから辿る `docs/...`」と参照し、skill 自身を冗長なリンク集にしない。
- skill **同梱**のファイル(`references/` 等)を bash 注入で参照するなら `${CLAUDE_SKILL_DIR}` を使う(cwd に依らず skill ディレクトリへ解決される)。

(人間や markdown ビューアが**ファイルとして**読む `docs/**` 同士のリンクは、通常どおり相対リンクでよい。上記は実行時に展開される skill 本文に限った注意である。)

## 5. ハーネスは「読まれる仕組み」まで作る(完了の定義)

参照ドキュメントや skill を新設・変更したら、**それが必要なときに実際に読まれるところまで用意して初めて「完了」**とする。
「詳細はここ」と受動的なポインタを置くだけでは、いつ読むか・読まねばならないかが曖昧で、結局読まれない。
次をすべて満たす。

1. **いつ読む/読まねばならないかを明示する。** 何の作業のときに必須なのかを、ドキュメント冒頭と索引([README.md](README.md))に書く。
2. **索引に登録する。** [README.md](README.md) の一覧表に、ドキュメントと「読むべきタイミング」を追加する。一覧から漏れた文書は存在しないのと同じになる。
3. **読まれる仕組み(トリガー)を用意する。** 受動的なポインタに頼らず、次のいずれかで発火させる。
   - **作業内容で発火する規約**(Issue/PR、design doc、テスト戦略など)→ その作業を `description` でトリガーする **skill**(`/pr-workflow`・`/design-doc` 系など)から参照させる。skill 本文は起動時にコンテキストへ読み込まれる。
   - **特定のファイル操作で発火する規約**(ハーネス編集など)→ **フック**でリマインダーを注入する。例: ハーネスのファイルを Edit/Write すると PreToolUse フック [`harness-edit-guard.mjs`](../../scripts/hooks/harness-edit-guard.mjs) が editing.md と索引を読むよう促す(`.claude/settings.json` に登録)。
   - **事実上どの作業でも必要なゲート**だけ CLAUDE.md(常時ロード)に置く。
4. **発火を確認する。** フックなら実際に動かして注入されること、skill なら description が想定タスクでトリガーすることを確かめる。

ドキュメントやスキルを足す変更には「正しく使われるようにセットアップする」ところまでを必ず含める。
セットアップ(索引登録・トリガー・発火確認)を伴わない規約追加は未完成とみなす。

## 6. SKILL.md は「指示書」— 実行することだけを書く

`SKILL.md`(スキル本体)は、起動したエージェントに **何をどう実行してほしいかの指示** を書く文書である。
一般的な解説ドキュメントではない。
次を SKILL.md に書かない。

- **設計判断の理由・正当化。** 「なぜこの方式にしたか」「なぜ別案でなくこれか」「移植性のため」といった、そのスキルをそう作った理由は SKILL.md に書かない(design doc / PR / コミットに書く)。SKILL.md には結論=「こう実行する」だけを書く。
- **過去の状態・変更の経緯・差分。** 「以前は X だった」「当初は Y を使ったが Z に変えた」は書かない。現在の指示だけを書く(履歴は Git が持つ)。これは SKILL.md に限らず、現状を説明する**すべてのハーネス文書**(`docs/harness/` 等)に共通する。
- **実行に不要な関連情報の寄せ集め。** 別スキルの使い分け説明など、いまそのスキルを実行するのに要らない情報を混ぜない(例: あるスキルの指示の中で、別スキルをいつ使うかを併記しない)。
- **書式・フォーマットの指示は具体例で示す。** 抽象的な説明でなく実際の記述例を載せる(「N と回答内容」ではなく「`4: 実際の回答をここに書く`」のように)。

迷ったら「これはこのスキルを実行するエージェントが、実行のために読む必要があるか」を問う。
理由・経緯・他スキルの紹介・抽象説明だけ、はいずれも不要。
なお、役割が近いスキルを作り分ける前に、**既存スキルに文脈を足して使えば足りないか**を必ず検討する(分ける意味が無ければ分けない。分けるなら各スキルの `description` でどちらを使うかが一意に分かるようにする)。

## チェックリスト(ハーネス編集を終える前に)

- 加筆内容が既存のどこかと重複していないか。重複は1か所に統合したか。
- 追記後の文書は、継ぎ足しの層ではなく一貫した1本の文書として読めるか。
- CLAUDE.md に足したものは、本当に常時必要な standing 規約か(条件付きなら外出ししたか)。
- 新設/変更したドキュメント・skill を [README.md](README.md) の索引に「いつ読むか」付きで登録したか。
- 受動的なポインタで終わらせず、読まれる仕組み(skill の description / フック / CLAUDE.md のゲート)を用意し、発火を確認したか(§5)。
- SKILL.md に、設計理由・過去の状態/差分・他スキルの紹介など実行に不要な情報を書いていないか。書式の指示は具体例で示したか(§6)。役割が近いスキルを新設する前に、既存スキル + 文脈で足りないかを検討したか。
- 誤読の余地がなく一意に読めるか。
