# orvk Design Docs

orvk の設計判断を意思決定の記録(decision record)として残すフォルダ。
「いまの設計がどうなっているか」だけでなく、「そのときどう意思決定したか」「どの代替案を比較してなぜ捨てたか」を時系列で読めることに価値を置く。

## ルール

### 対象はライブラリ側の設計判断のみ

design doc に書くのは orvk ライブラリ本体(crate 本体、ビルド基盤)の設計判断である。
examples やサンプルコード自体の設計判断は design doc に書かない。
サンプルの修正がライブラリ側の変更を必要とする場合、その**ライブラリ側の変更**だけを design doc にする。

### design doc に書くのは「覆す代償が大きい決定」

何を design doc に残すかは「間違えたときの代償」で測る。
実装後に変えるのが難しい・高くつく決定(言語や依存ライブラリの選択、アーキテクチャの方針、API の骨格、handle や同期の契約など)は doc に書いて意思決定を残す。
数時間で容易に覆せる些末な決定(内部変数の命名、ログメッセージの文言、エラーメッセージの見せ方など)は doc に書かない。
すべてを書き切ろうとすると設計段階で実装を書くことになり、design doc の意味が失われる。

### design doc は design の高度で書く(実装の詳細は書かない)

「覆す代償が大きい決定」を doc に書くと決めた後も、その doc を **どの高度(altitude)で書くか** を間違えてはいけない。
design doc が記録するのは **設計の意思決定とデータ契約** であって、実装そのものではない。

> If you specify every possible detail in a design doc, you've essentially written the implementation during the design phase. That would defeat the whole purpose of a design doc.([Write an Effective Design Doc](https://refactoringenglish.com/excerpts/write-an-effective-design-doc/))

実装まで書き切ると、設計段階で実装を書いてしまい、design doc の意味が失われる。
ここで分けるのは「doc に書く粒度」と「執筆時に検証しておく粒度」である。
設計がコードとして成立すること(提案する型や分割でコンパイルが通ること、コードレベルの整合)は、doc には書かないが執筆時に必ず検証する。
承認後の実装で「この設計ではコンパイルが通らない」と判明する手戻りは、design doc が最も避けるべきものだからである。
そのうえで、検証済みの実装レベルの詳細(関数の切り方、呼び出し順)は doc に書き写さず、コードそのものの正しさの確認は PR(コード)レビューが担う。
design doc のレビューは「この設計の筋が正しいか・取り違えると大きな手戻りになる判断が妥当か」を見る場である。

**design doc に書く(design 高度):**

- モジュール・責務の境界、phase / lifecycle の構造、状態とデータの流れ。
- データ契約・型の形(何の情報が、どんな形で受け渡されるか)。返り値型や主要な struct の中身は、それが無いと設計が伝わらないなら書いてよい(むしろ書くべき)。
- API サーフェス(利用者が触る口)とその意味論。
- 以上に関する意思決定の根拠・トレードオフ・代替案。

**design doc に書かない(実装高度。PR レビューで扱う):**

- 具体的な関数名・関数分割、どの関数がどの関数を呼ぶか、呼び出し順序・手続きのステップ列。
- 述語・分岐ごとの編集レシピ(「この関数のこの match にこの variant を足す」式の逐次手順)。
- 変更するファイル・関数のインベントリ(「次のファイルを更新する」一覧)。
- 改名の機微、どのテストが何をカバーするか、といったコード作業の段取り。

コードや型を載せてよいのは、それが **意思決定やデータ契約を伝える**ときに限る(実装スクリプトとしてではない)。
判断テストは「この記述を取り違えると大きな手戻りになるか? それとも PR レビューで気づいて直せるか?」である。
PR レビューで検証・修正できることは doc に書かず、設計判断とデータ契約だけを残す。
詳細設計が目指すのは「同じ実装が再現できる詳細」ではなく「この設計に同意できる詳細」である。
実装可能性の検証(関数の切り分けでジェネリクス込みにコンパイルが通るか等)は執筆時に必ず行い、その検証過程は doc に書き写さない。

ただし、ある不変条件が複数のコード箇所で同じ判定を重複させる形になるときは、**その単一の分類点・出所をどこに置くか**(状態経路の一本化)は design 判断なので doc で決める。
関数分割そのものは PR レビューに委ねてよいが、「単一出所を1つ置くか、分散したまま convention + 検知点で受けるか」は、取り違えると複数箇所が歩調を崩して drift する手戻りの大きい判断なので、design 高度で決めて根拠を書く。

### 「負荷・コスト」はランタイムの負荷であって実装工数ではない

「負荷・コスト」セクションに書くのは、この設計が **ランタイム・運用で背負うコスト**(submit ごと・frame ごとの CPU / GPU 仕事、メモリ、アロケーション、データ量に対するスケール挙動など)と、それが設計の妥当性に与えるトレードオフである。
**実装の工数・実装範囲(何ファイル・何関数を変えるか、難易度、スケジュール)はここに書かない。**
それは設計の妥当性の判断材料ではないし、変更ファイルの列挙は上の「design の高度で書く」が禁じる実装高度の記述でもある。
考慮不要なら、なぜ不要か(例:「この設計は hot path を増やさない」)を一文書く。
負荷・性能の主張は可能な限り測定可能・反証可能な形で書く(「十分速い」「上乗せは小さい」だけで済ませず、何に比例するか・hot path に何を足すか・どの条件で成り立つかを書く)。

### 1ファイル = 1つの意思決定

design doc は「その意思決定をレビューできる単位」で作る。
全体アーキテクチャやモジュール構造のような広いスコープの doc から、1つの細かい要素についての doc まで、スコープの大きさは様々でよい。
重要なのは、レビューする人が「この決定に同意するか」を判断できる単位で切られていることである。

### Design Doc は「何のためにどんな設計か」と「その妥当性の判断根拠」を主張する文書(簡潔に書く)

design doc は **(1) 何のためにどのような設計を入れるのか** と、**(2) その目的・設計が妥当だとレビュアーが判断できる根拠** の両方を主張する文書である。
design doc はユーザーのレビュー単位でもあり、(2)——目的と設計の妥当性を検証するために要る情報——が本文に無ければレビューできない。
(1)(2) の両方が核であり、読み手はまず核をつかんでから詳細を読む。
だから次を守る。

- **概要は 10〜20 行に収め、「何のために・どのような設計を入れるのか」がそれだけで読み取れる**ようにする。概要にコードの詳細を持ち込まない(詳細は「詳細設計」に書く)。概要がこの分量に収まらないなら、1 つの design doc に詰め込みすぎている兆候なので、分割すべきかを疑う。
- **妥当性の判断根拠を必ず含める。これは冗長ではなく必要な情報である。** なぜこの設計が目的に対して妥当か、検討した代替案と Pros/Cons・トレードオフ、既知の落とし穴・負荷/コストを書く(代替案と「なぜメイン案を選んだか」は核。後述「テンプレート」を参照)。これらはレビュアーが「この決定に同意するか」を判断するのに要る情報なので、簡潔さを理由に削らない。
- **(1)(2) を補強しない文章は入れない。** 主張にも妥当性判断にも寄与しない水増しで核をぼやけさせない。テンプレートのセクションを機械的に埋めること自体は目的ではない。判断基準は「読み手が『何のために・どんな設計を・なぜそれが妥当と言えるか』を筋の通った形で読み取れるか」。削ってよいのは寄与しない水増しだけで、(2) の判断根拠は削らない。
- **同じ根拠を複数セクションで繰り返さない。** 同じトレードオフ・根拠・概念説明は1つのホームセクションに集約し、他のセクションは結論一文+そのホームへのポインタにする。概要・やらないこと・シナリオ・落とし穴・詳細設計で同じ説明を何度も書くと、(2) の判断根拠を新規に足さない再陳述で核がぼやける。これは「正確だが重複した」水増しで、情報ゼロの水増しと同じく削る対象である。
- 用語は初出の本文で **インライン定義する**のを基本とし(読者を用語集へ飛ばさず、その場で分かるようにする)、doc 全体で何度も使う独自用語・内部名・略語だけを独立した **用語集** セクションにまとめてよい。詳細設計の中だけで使うコードの型などは、用語集に出さず詳細設計内で説明する。詳細設計のサブセクションが多いなら、各セクションが何を説明するかの **1〜3 行サマリー(目次)** を詳細設計の冒頭に置き、ポインタにしてよい。

### 初稿は執筆側が完成させる — 完成前にユーザー判断を仰がない

ユーザーのレビュー単位は **書き上がった design doc(初稿)** である。
**完成した初稿を人間がレビューする**のであって、初稿を作る過程でユーザーに設計判断をさせるのではない。
「最初から reject されない完璧な doc」にしようと完成前にユーザー判断を仰ぐのは、この運用を無意味にする誤りである。
ユーザーとのやりとりには時期で線を引く。

- **執筆前(何を書くか)はすり合わせてよい。** そもそも何を目的とし、どの問題を解くべきか、スコープの外枠はどこか、というユーザーの問題意識は、執筆を始める前にすり合わせる。問題設定が曖昧なら `grilling` で1問ずつ引き出してよい。エージェントが「何を書くべきか」を理解しないまま書き始めない。これは doc の前提を固める正当な作業である。
- **初稿の設計判断は、すべて執筆側が決めて書き切る。** いったん書き始めたら、設計上のあらゆる選択を **あるべきコードの姿・哲学(docs/philosophy.md)・各案の Pros/Cons・(補助的に)既存実装との一貫性** から自分で決める。これには採用案・却下案だけでなく、**doc の分割・粒度・スコープ・番号体系をどうするか**(1 つに収めるか複数に割るか、どこで境界を引くか)も含む。分割の可否はユーザーの仕事ではなく、初稿を作る執筆側の仕事である。迷うほど有望だった候補は **代替案** に Pros/Cons とともに残し、最良のものを本命として書く。
- **完成前にユーザーへ設計判断を投げない。** 個別の設計判断・分割・スコープ・機構の選択を、初稿完成前に `AskUserQuestion` 等でユーザーに尋ねない。敵対的レビューで major が出ても(収束しなくても)、それを執筆側が決めて初稿を仕上げる責任は執筆側にある(「非収束だからユーザーに問う」は誤り。下記「運用の流れ」「design-doc スキル」)。設計が根本的に立たない(reject 相当)と判断したときも、「どうしますか」と問うのではなく、**reject 推奨とその理由を決めた立場として doc に記録して初稿を仕上げ**、最終 approve / reject の確定だけをユーザーに委ねる。

こうして一貫性のある筋の通った初稿を執筆側が書き上げてから develop に landing し、ユーザーが doc 単位でレビューする(「運用の流れ」を参照)。

### 関連する design doc はまとめてレビューする

1つの設計テーマを複数の design doc に分けて記録することがある(密結合な設計セット)。
そうした **関連する doc は、個別にバラバラにレビュー・修正しない。セットでまとめてレビューする。**
「1ファイル=1意思決定」は記録の単位であって、レビューの単位を1ファイルに縛るものではない。
個別に見ると「現状の分割を所与」として各 doc 内だけを評価してしまい、次を扱えなくなる。

- **分割の是非と粒度**: そもそも分割すべきか。境界の引き方は妥当か。統合・再分割した方がよくないか。
- **doc 間の整合性**: 共有する概念・型・契約の説明が doc 間で食い違っていないか。境界に隙間(どの doc も書いていない)や重複(複数 doc が別々に書いている)がないか。
- **責務分担を踏まえた執筆**: 各 doc が他の doc の担当範囲を前提に、過不足なく書けているか。

ある doc が他の(とくに未承認 draft の)doc を相互参照している、または同じ設計テーマを分担しているときは、それらを1つのセットとして同時にレビューし、修正もセット内の整合を取りながら行う。
どこまでを1セットにするかは、相互参照と設計テーマで判断する(独立した doc は単体でレビューしてよい)。

ただし、**レビュー単位の運用判断は doc 本文に書かない。** 「この doc を ○○ と同一 PR に束ねるべきか」「独立にレビューできる」といった束ね方の判断は、レビュー時にユーザーが決めることである。doc 本文でそれを正当化すると、(2) の設計妥当性ではなく運用の自己弁護で水増しになり、しかも上記の既定(関連 doc はまとめてレビュー)と逆向きの主張を doc 側で抱えることになる。関連 doc については、**技術的依存の事実**(どちらがどちらの前提をどう更新するか、どの軸で直交するかの技術的根拠)だけを本文に書く。

### 番号とファイル名

`NNNN_title.md` の形式で 4 桁の連番を振る。
番号は意思決定が行われた順(古い → 新しい)。
前から番号順に読んでいくと、どういう順番で設計判断が行われ、実装され、変わっていったかが分かるようになっているべきである。

- 古い番号の design doc への参照は自由。
- 新しい番号の design doc への参照は、同時に複数の design doc をまとめて作成したときなどに限られるはずである(時系列が逆転するため)。

### 日付と不変性

ヘッダーに `created`(執筆日)を必ず入れる。
意思決定の時期そのものが情報である。
大きく内容を更新した場合は `updated` に更新日を書いてよい。

**`updated` に書くのは日付だけである。** 何を変えたか(「レビュー反映」「全面改稿」「実装完了」など)の変更内容・改稿理由をここに書かない。
変更履歴は git が持つのが本リポジトリの前提であり、`updated` の後ろに変更説明を足すのは、上述の「沿革・時制の物語」を本文に持ち込むのと同じ残骸である。
何を変えたかを残したいなら、それはコミットメッセージに書く(doc 本文には残さない)。
複数回更新したときは日付を最新の1つに更新し、日付だけの `updated` 行を積み増さない。

**昔の design doc を消して書き直さない。**
設計を変えるときは新しい design doc を新しい番号で作成し、「なぜ設計を変えるのか」「どのような設計案があったか」「なぜその案を選んだのか」をその doc 本文だけで分かるように記録する。
関連する番号付き design doc へのリンクは置いてよいが、読者が別 doc を開かないと本文を理解できない書き方にしない。
元の doc はそのまま残す(必要ならヘッダーに `superseded-by: NNNN` を追記してよい)。

### status

- `draft` — 本当に書きかけ(未完成)。別のエージェントに執筆を引き継ぐとき等に使う。
- `ready for review` — 完成し、ユーザーのレビューを待つ段階。
- `reviewing` — レビュー PR でレビュー中(下記「レビュープロセス」)。
- `approved` — ユーザーがレビュー PR で明示的に承認した設計。
- `ai-approved` — コーディングエージェントが任された範囲で自分の責任で判断・実装した設計(ユーザーのレビューを経ない委任判断)。ユーザーの明示承認とは区別する。
- `rejected` — 検討の結果、採用しなかった設計。記録として残す。

**レビュー前の doc も develop に集約する。** `draft` と `ready for review` は、いったん develop に landing する(下記「運用の流れ」)。ユーザーは develop 上の markdown をビューア等でブランチ切替なしに読めるべきで、レビュー前の doc を feature ブランチに散逸させない。
ステータスの遷移とレビューの進め方は下記「レビュープロセス」を参照する。
実装まで任されて行った委任判断は `ai-approved`、レビュー PR でユーザーが承認した設計は `approved` とする。

### reject 時の却下理由セクション

design doc を `rejected` にするときは、なぜ採用しなかったかを `## 却下理由` という独立したセクションに書く。
場所はヘッダー(メタデータ)の直後、`## 解決したい問題` の前に置く。
読者が冒頭で「この doc は却下された」「なぜ却下されたか」を把握できるようにする。

- 引用ブロック(`>`)や本文冒頭の地の文で reject 理由を書かない。reject 理由は doc の構成要素なので、見出し付きの正式なセクション(`## 却下理由`)にする。
- `## 却下理由` には、提案が採用されなかったこと、却下の理由(設計上どこが不十分だったか)、代替の設計があれば `superseded-by` の doc へのリンクを書く。
- ワークフロー上自明なメタ説明を書かない。doc を消さず残すのは規約で決まっているので、「規約により記録として残す」のような、規約どおりに運用していること自体の説明を doc 本文に書かない。doc には設計と却下理由という中身だけを書く。
- 却下された doc の本文(解決したい問題〜代替案)はそのまま残す。書き換えない。

### implementation(実装状態)

承認(status)と実装の完了は別の軸なので、ヘッダーに `implementation` を併記する。

- `not-started` — 承認済みだが実装着手前。
- `in-progress` — 実装中。
- `done (YYYY-MM-DD)` — 実装完了。完了日を必ず記録する。

### 自己完結の原則

design doc は単体で読めて完結していなければならない。
参照してよいのは**他の design doc と恒久的な文書(docs/philosophy.md 等)だけ**。
削除済み文書、吸収済み文書、フィードバック md などの一時的な文書、会話ログ、編集前の design doc の状態への参照を作らない。
背景説明に必要な内容は、参照ではなく design doc 本文に書き写す。

採用しない案はすべて「代替案」セクションに書く。
「解決したい問題」「概要」「詳細設計」では、現在採用する設計とその必要性を書く。
過去の編集状態や、採用しない案を本文の前提として説明してはいけない。

「やらないこと」と「代替案」は、それぞれ self-contained に書く。
一行の案名と Pros/Cons だけで済ませない。
読者がその design doc だけを読んで、次の内容を理解できる粒度で書く。

- その案や禁止事項が具体的にどういう設計なのか。
- その案が何を解こうとしているのか。
- なぜ採用しないのか、またはなぜ今後もやるべきではないのか。
- 永続的にやらないのか、特定の条件なら再検討しうるのか。

悪い例:

```md
また、前案の fluent API は次のような形だった。
```

```md
以前の版では access 宣言がコマンド記録の直後に続く設計だったが、今回は変更する。
```

```md
詳細は削除済みの api_feedback_round3.md の B-2 を参照。
```

```md
- staging buffer の自動 flush を持たせない。
```

良い例:

```md
access 宣言 API は、宣言とタスクの記録がコード上で同じ値に閉じている必要がある。
呼び出し順に依存する builder は、宣言がどのタスクに属するかを読み取りにくくする。
```

```md
## 代替案

- `.uses(handle)` の後に `.access(...)` を続ける順序依存 fluent API を採用する。
  - Pros: 書く文字数は少ない。
  - Cons: access 宣言の所属が builder の内部状態に依存する。
```

```md
- staging buffer の自動 flush を持たせない。

  自動 flush は submit 時点の暗黙の書き込みを増やし、記録した内容と実際に GPU へ渡る内容の
  対応を読みにくくする。flush が必要な箇所を利用者が明示的に呼ぶ方が、転送のタイミングを
  API 上で追跡できる。暗黙の便宜が欲しい場合も、明示 API の上に helper として足せば十分である。
```

### 却下・不在の概念を本文に残さない(残骸の禁止)

design doc は「いま採用する設計」を、それ自体で完結した正の記述として書く。
**リポジトリに一度も入っておらず、レビューで却下されて入らないと決まった概念**(前任 draft が足そうとして却下された API、廃案になった型名など)を、本文に「比較対象」や「削除した事実」として残してはいけない。
読者にとってその概念は存在しないので、不在をわざわざ記録すると混乱とノイズになるだけである。

具体的な禁止:

- **否定・対比表現を本文(解決したい問題 / 問題の背景 / 概要 / 詳細設計 / 用語)に書かない。** 「X ではない」「X は作らない」「X と違って」「以前は X だった」「旧 X」のような、読者が見たことのない X を前提にした対比は本文に置かない。採用する設計を、X を引き合いに出さずに肯定文で説明できるはずである。
- **却下案・不採用方針は「代替案」または「やらないこと」にだけ書く。** しかもそれは「実際に設計選択肢として比較検討したもの」に限る。前任 draft の誤りの訂正(「前の版は X を足していたが消した」)としては書かない。代替案には「その案はどういう設計か / 何を解くか / なぜ採らないか」を self-contained に書き、X が却下された経緯ではなく X 自体の Pros/Cons を書く。
- **存在する要素の削除・改名は別。** 既にリポジトリにある型・API を今回の変更で消す/改名する場合、その削除対象を列挙するのは正当(読者はその要素を知っている)。禁止されるのは「リポジトリに存在しない幻の概念」への言及である。

悪い例(リポジトリに `BufferConfig` / `ResourceConfig` trait が存在しないのに本文で言及):

```md
buffer 生成の設定は Config ではない。`ResourceConfig` trait は作らない。
```

```md
以前の draft では設定を config 構造体に入れていたが、今回はビルダーにする。
```

良い例(採用案を肯定文で説明し、却下案は代替案へ):

```md
required な値は constructor 引数で受け取り、optional な設定は
`.usage(...)` のようなビルダーメソッドで受け取る。
```

```md
## 代替案

- config 構造体に全フィールドを集めて `Buffer::new(BufferConfig { .. })` で作る案。
  - これは required field を struct field として表せる。
  - しかし optional な設定まで struct に入り、既定値との使い分けが冗長になる。
    ビルダーで optional を表す方が simple なので採らない。
```

### 変更対象の既存実装は「なぜそうなっているか」まで本文に書く

design doc が **既存の実装・挙動・契約・ビルド基盤を変更する**提案(「現状の実装には問題がある」「いまの設計を別の形に置き換える」「既存の挙動からこの抽象を切り出す」「既存の API・挙動を後継を立てずに削除・廃止する」「既存のデフォルト値・不変条件・契約を変える」など)を含むとき、その既存実装が **何をしているか** だけでなく **なぜそうなっているか**(どんなドメイン上の要請やシステム・実装上の制約がその形を必要としているか)を、本文に self-contained に書く。
ここでいう既存実装にはコードだけでなく、既存のデータ契約やビルド基盤も含む。

ただしこのルールが発火するのは、**既存挙動の存在理由が読者にとって非自明で、それを知らないとこの変更の妥当性を評価できない場合**に限る。
既存挙動が素直・自明で、現状を一行示せば読者が変更の是非を判断できる場合は、背景の追加記述は要らない。
既存挙動を据え置いたまま新しい口やフィールドを足すだけの additive な doc(デフォルト経路は不変)も、その既存挙動はこのルールの対象外である。
対象は読者に「現状 → 新設計」の置換の是非を問う doc に限る。
なお、後継を立てずに削除・廃止する doc も対象である。
削除対象を列挙してよいこと(隣の「残骸の禁止」の carve-out)と、その要素が現状なぜ存在するか(どんな要請を満たしていたか)の説明を省いてよいことは別である。

design doc は全文脈を把握していない読者が読む。
読者は既存コードを知らないので、「現状はこうなっている」と現象だけ示されても、それが **必要があってそうなっているのか、直すべき不備なのか**を区別できない。
とくに、一見すると不自然・不要に見える既存の構造(例: 実際には GPU へコマンドを submit しないのに実行される記録処理)は、文脈を持たない読者にとっては存在自体が謎である。
既存実装の存在理由が書かれていなければ、読者は置換が妥当かを判断できず、新設計のために導入する抽象(新しい trait・phase・型など)が **なぜ要るのか** も判断できない。

満たすべき内容は次の 2 つである。
両方が必要で、**目的の提示だけでは足りない**。

- **その形を強いている制約**(なぜ素直な形にできないのか)を書く。「〜のために呼ばれている」と目的を述べるだけでは満たさない。後述の悪い例も目的句(「計画を作るために」)は含んでおり、目的の有無は合否を分けない。良い例との差は、不自然な形を強いている制約(計画に必要な出力情報を現状その経路を通さないと得られない、など)を述べているか否かである。
- **存在理由を問題列挙より先に置き、後続の問題がその帰結として読めるように**書く。問題を述べた後に装飾的な一文を後付けするのではなく、存在理由を grounding として先に示す。

なお、既存挙動に積極的な設計理由が無い(単なる既定値や、特段の経緯のない素直な実装)場合は、その旨をそのまま書けばよい。
無理にドメイン理由を捻り出す必要はない。
「現状は素直に既定を使っているだけで特別な理由は無い」と現状を肯定文で示せれば、それで足りる。

書くのは **この変更の妥当性を評価するのに必要な背景に限る。**
コードの全履歴や変更の経緯(いつ誰が、といった沿革)、関係ない既存実装の網羅的な解説までは要らない。
判断基準は「読者がこの設計変更に同意するかを、この doc だけで判断できるだけの背景があるか」であり、置き場所は問わない(「問題の背景」が自然だが、「解決したい問題」や代替案など doc 内のどこに書いてもよい。要件は配置ではなく、存在理由が doc 本文だけで読み取れることである)。
ここでいう self-contained の単位は**この doc 単体**である。
複数 doc を同時に作成する場合でも、変更対象の既存実装の存在理由は、その変更を含む doc 自身の本文に書く。
前提を兄弟 doc に置いて参照で済ませない(自己完結の原則を参照)。

これは「却下・不在の概念を本文に残さない」と矛盾しない。
あちらが禁じるのは **リポジトリに存在しない幻の概念** を引き合いに出す否定・対比である。
こちらが求めるのは **実在する既存実装** の存在理由の説明であり、あちらの carve-out(「存在する要素の削除・改名は別。読者はその要素を知っている」)にそのまま当たる。
存在理由は「現状こういう制約からこの形になっている」と **現在形・肯定文** で書く。
「以前は X だった」「旧 X から変えた」といった沿革・時制の物語は、ここではなく(該当すれば)代替案で扱う(残骸の禁止の対象でもある)。

悪い例(現象と目的だけ示し、その形を強いている制約=存在理由を書いていない):

```md
現行実装では、barrier の計画を作るためにも `record` が呼ばれている。
この構造には次の問題がある。

- `record` が「実行される command 記録」と「計画のための access 抽出」の
  2 つの意味を持っている。
```

(目的「計画を作るために」は書かれているが、なぜ計画段階で `record` を通さざるを得ないのかという制約が無い。読者は「コマンドを捨てる record 実行がなぜ存在するのか」を知らないため、この変更の是非を判断できない。)

良い例(その形を強いている制約=存在理由を先に説明し、問題をその帰結として述べる):

```md
タスクグラフは、実際に GPU へ submit する前に barrier の計画を立てる必要がある。
計画には各タスクが実際に触るリソースの access 集合が要るが、その access は現状
`record` の実行中にしか宣言されない。そのため現状は計画段階でも、実 submit を
伴わないまま `record` をコマンドを捨てる encoder に対して呼び、access 宣言だけを
取り出している(実行されない record が存在するのはこのためである)。この二重の
意味づけが次の問題を生む。

- `record` が「実行される command 記録」と「計画のための access 抽出」の
  2 つの意味を持っている。
```

### 図(mermaid 等)は単体で誤読なく伝わること

図を載せるときは、その図だけを見て——doc の文脈を共有していない読者が見ても——何を表しているかが誤読の余地なく伝わるようにする。
design doc を読む人は執筆者と同じ文脈を持っていない。
文脈を共有していない人が読んでも読み方に困らない図にする。

- フローチャートや依存ツリーなどの矢印は、一つの図の中で意味を統一する。同じ図で矢印が「依存」「データの流れ」「生成する」「使う」など複数の意味で混在してはいけない。意味が混ざると、読者は矢印ごとに意味を推測することになる。
- 標準的でない図(独自のまとめ方、矢印やノードの意味が一目で定まらない図)を載せる場合は、その図の読み方(凡例: ノードが何を表すか、矢印が何を意味するか)を本文か図の近くに必ず添える。読み方の説明が要る図を、説明なしで単体で置かない。
- 目指すのは「なんとなく分かった気になる」「たぶんこういう意味だろうと当てずっぽうで推測できる」図ではなく、言いたいことが誤読の可能性なく伝わる図である。読み方の説明が要るほど複雑な図は、そもそも図を分割するか、図をやめて文章にすることを先に検討する。説明してもなお複数の読み方ができる図は載せない。

### テンプレート

[template.md](template.md) を土台にする。
完全にテンプレートに沿う必要はなく、必要に応じて章の追加・削除をしてよい。
ただしテンプレートの各セクションに書かれている説明文(プレースホルダ)は必ず実内容に差し替え、完成した design doc に残さないこと。

「代替案」と「なぜメイン案を選んだか」は design doc の核なので省略しない。
最初から1案しか考えていなかった場合でも、後から見る人のために他の選択肢とその Pros/Cons を書き起こす。

#### 代替案はレビュー中に詳しく、approve 後に簡潔へ

最終的な doc では、見送った案は簡潔な説明と見送り理由で足りる(網羅的に書かない)。
ただしレビュー中に、本命と肩を並べる魅力的な代替案があってユーザーにどちらを選ぶか問いたい場合は、両案を **同じ粒度** で詳しく書いて比較できるようにする。
approve 後は、選ばれなかった案を簡潔な説明 + 見送り理由に整理し直す。
詳しい比較は approve 前のコミットとして git 履歴に残るので、最終 doc は読みやすい分量に収まる。
この approve 後の整理は、レビュー PR をマージする前(または status を `approved` にする変更)で行う。

### 運用の流れ

design doc は **書き上げて develop に集約し、ユーザーがレビューするときに PR を立てる**。
1 doc(または密結合な 1 設計セット)= 1 つの長命 `feature/design-<topic>` ブランチで書く。

1. 設計が必要な作業の前に design doc を書く。書きかけは `draft`、完成してレビューを待てる状態になったら `ready for review` にする。
2. **`draft` / `ready for review` のどちらも、いったん develop に landing する**(`feature/design-<topic> → develop` を `--no-ff`、PR なし)。これでレビュー前の doc が develop に集まり、ユーザーはブランチ切替なしに読める。フィードバックや続きの執筆は同じブランチに追記して再度 develop へマージする。
3. ユーザーがレビュー開始を宣言したら、develop から対象 doc の status を `ready for review` → `reviewing` に変える **レビュー PR** を立てる(`/pr-workflow`「Design Doc レビュー PR」)。関連 doc があれば1つの PR にまとめる(下記「レビュープロセス」)。
4. レビューはその PR のコメントで行い、各スレッドの解決を doc 本文に反映する(PR ブランチに追記コミット)。
5. ユーザーが PR で承認したら status を `approved`(代替案を簡潔形に整理。「テンプレート」)、設計が立たないと結論したら `rejected`(`## 却下理由` 節を付ける)にして、ユーザーが PR を develop にマージする。
6. 承認後の実装は [../harness/git-and-pr.md](../harness/git-and-pr.md)「Design Doc に紐づく実装」に従う。
7. 後日設計を変えるときは新しい番号の doc で上書きの意思決定を記録する。

エージェントが PR なしで develop にマージしてよいのは、上記 2 の **レビュー前 doc(`draft` / `ready for review`)の landing** に限る。
レビュー PR(`reviewing` 以降)を含むそれ以外の develop / main へのマージはユーザーが行う。
詳細は [../harness/git-and-pr.md](../harness/git-and-pr.md)「Design Doc のブランチ運用」を参照。

### レビュープロセス

レビューは GitHub の PR 上で行う(Google Docs 等は使わない)。
レビュー対象の doc は既に develop にあるので、**レビュー PR は develop から status を `reviewing` に変えるだけ**でよい。GitHub は変更ファイルなら差分の外の行にもコメントできるので、status 一行だけの差分でも全文をレビューできる。

- **PR の単位**: 1 doc、または相互参照する/同じ設計テーマを分担する関連 doc のセット。PR を作る前に、同時に提示すべき関連 doc が無いか必ず確認し、あればまとめて `reviewing` にして1つの PR に入れる(「関連する design doc はまとめてレビューする」)。
- **レビュー開始は skill で**: ユーザーがレビュー開始を宣言したら、対象を `reviewing` にしてレビュー PR を作り URL を提示する(`/pr-workflow`「Design Doc レビュー PR」)。
- **コメントは PR で、解決は本文へ**: 議論は PR のスレッドで行い、各スレッドが収束したら **その理解を doc 本文へ反映する**。本文を読めば分かる状態にし、PR コメントの中だけで説明を完結させない(out-of-band にしない)。
- **スレッドを放置しない**: 対応済みのスレッドは resolved にする。長引く論点も、まず本文の決定として書き切ることを目指す。「未解決の論点」節へ移せるのは、先送り自体を設計判断として根拠付けられる場合だけである(節に書くべき内容は template.md)。
- **確定**: 一通り問題が無くなりユーザーが承認を投稿したら `approved`、根本的に立たない/不要と結論したら `rejected` にして、ユーザーが PR を develop にマージする。

### レビューしやすく書く

レビュアー(執筆者の文脈を共有しない人)が短時間で評価できるように書く。

- **冒頭を全読者に分かるものにする**: goals と具体的なシナリオを前に置き、早い段落ほど誰でも読めるようにする。
- **専門用語を乱用しない**: 専門用語は、より一般的な語に置き換えるか初出でインライン定義する(「主張する文書」「用語集」を参照)。
- **図に投資する**: 構成要素の関係・データの流れ・相互作用・複雑な手順は図にする(「図は単体で誤読なく伝わること」)。
- レビューには十分な時間を見込み、まず一人(ここではユーザー)のレビューで粗を取る。
