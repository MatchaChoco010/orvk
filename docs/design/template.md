# Title of Design Doc

- created: 20XX-XX-XX
- status: draft / ready for review / reviewing / approved / ai-approved / rejected
- implementation: not-started / in-progress / done (20XX-XX-XX)

各セクションは必要に応じて追加・削除してよい(README「テンプレート」)。
「(省略可)」と付いたセクションは、不要なら設けない・削除する。
ただし「概要」「詳細設計」「代替案」「なぜメイン案か」は核なので省略しない。

## 解決したい問題

どういった問題を解決しようとしているのかを簡潔に書く。
ユーザーの利益に直接つながらない技術的な問題を書いてもよい。
ここで「何を目的とするか」が読み手に一文で伝わるようにする。
目的は実装手段ではなく効果で書く(「X を導入する」ではなく「Y という問題がこう解消される」)。

## 問題の背景

なぜこの問題をいま解決しないといけないのかを簡潔に書く。
関連するコードや、仕様書、議論などの資料がある場合は必ず記載する。
既存の実装・挙動を変更する提案で、その既存挙動の存在理由が読者に非自明なときは、何をしているかだけでなく「なぜそうなっているか(その形を強いている制約)」も、文脈ゼロの読者が変更の妥当性を判断できるよう本文に書く(README「変更対象の既存実装は『なぜそうなっているか』まで本文に書く」)。

## この文書では書かないこと

いっけん関連しそうだが書かないと決めたことを列挙する。
書き忘れたのか書かないと意思決定したのかが読む人に伝わるようにすることが目的。
これを意識することで文章のスコープを明確にして design doc を書く必要がある。

- 書かないこと1
- 書かないこと2

## やらないこと

明らかにやらないことではなくて、やりうるかもしれないけど今はやらないという意思決定したことを列挙する。
これは feature creep とか早すぎる最適化とか、over-engineering を防ぐことが目的。
「この設計ではやらない」のか「将来 N ヶ月〜 N 年はやらない(やるべきではない)」なのかが分かるように書くこと。

- やらないこと1
- やらないこと2

## 用語集(省略可)

doc 全体で何度も使う独自用語・内部名・略語をまとめる。
用語は初出の本文でインラインに定義するのを基本とし、インライン定義で足りるならこのセクションは設けない。
doc をまたいで参照したくなる、または本文の複数箇所で前提になる用語だけをここに集める。
詳細設計の中でしか出てこないコードの型などは、ここではなく詳細設計の中で説明する。

## 概要

問題を解決するための設計の全体像を簡潔に書く。要約ではないことに注意。

**10〜20 行で「何のために・どのような設計を入れるのか」がそれだけでサクッと読み取れる**ようにする。
コードの詳細はここに持ち込まない(詳細は「詳細設計」に書く)。
この分量に収まらないなら、1 つの design doc に詰め込みすぎを疑う(README「Design Doc は『何のためにどんな設計か』と『その妥当性の判断根拠』を主張する文書」)。

全体を俯瞰できるように mermaid の図が1枚ぐらいあると嬉しい。
案が複数ある場合はメイン案を書き、他のは「代替案」に移動して読みやすくする。

## シナリオ / ユースケース(省略可)

設計が実際にどう使われるかを、具体的な利用シーンでなぞる。
とくに API 設計では、ライブラリを使ってレンダラーやアプリを書く人・ライブラリ自体を拡張する人などの立場で、書き味を具体例でトレースするとレビュアーが理解しやすい。
抽象的な説明より先に具体例を置くと、文脈を共有しない読者も設計の意図をつかめる。

## 詳細設計

概要で触れた全体像の各部分を、**design の高度**で深掘りして書く(README「design doc は design の高度で書く(実装は PR レビューへ)」)。
書くのは設計の意思決定とデータ契約(モジュール・責務の境界、phase / lifecycle 構造、状態とデータの流れ、型の形、API サーフェスとその意味論、そしてそれらの根拠・トレードオフ)である。
具体的な関数名・関数分割・呼び出し順序・分岐ごとの編集レシピ・変更ファイルや関数のインベントリ・どのテストが何をカバーするかは、実装高度なので**書かない**(PR レビューで扱う)。
コードや型を載せるのは、それが意思決定やデータ契約を伝えるとき(返り値型・主要 struct の中身など、無いと設計が伝わらないもの)に限る。

サブセクションが多い場合は、各サブセクションが何を説明するかの 1〜3 行サマリー(目次)を冒頭に置き、以降へのポインタにしてよい。
詳細設計の中だけで使うコードの型・関数などの用語は、ここで説明する(doc 全体で要る用語は上の「用語集」へ)。

解決したい問題に対して、短すぎると感じる場合は本当に design doc が必要なのか、長すぎると感じる場合は実装高度に降りていないか(同じ実装を再現する手順を書いていないか)を疑う。
各記述について「取り違えると大きな手戻りになるか、それとも PR レビューで気づいて直せるか」を問い、後者なら書かない。

設計について誤解なく全容が伝わるまでは精緻に書く。
ただし目指すのは「読んだ人が同じ実装を再現できる詳細」ではなく「読んだ人がこの設計に同意するか判断できる詳細」である。

## 落とし穴

設計上の既知の問題(「仕様」として受け入れる必要があるような若干のネガティブ要素や、拡張性の限界など)。

## 代替案

考えてみたけど見送った(でも議論の進みによっては再検討しうる)案を書く。
それぞれメイン案に対する Pros/Cons を書き、なぜメイン案を選んだのかも書く。

最終的な doc では、見送った案は簡潔な説明と見送り理由で足りる。
ただしレビュー中に、本命と肩を並べる魅力的な代替案があってユーザーにどちらを選ぶか問いたい場合は、両案を同じ粒度で詳しく書いて比較できるようにする(README「代替案はレビュー中に詳しく、approve 後に簡潔へ」)。
approve 後は、選ばれなかった案を簡潔な説明 + 見送り理由に整理する(詳しい比較は approve 前のコミットとして git 履歴に残る)。

## セキュリティ・プライバシー(省略可)

攻撃面・信頼境界・機微なデータの扱い(保持・アクセス制御・暗号化)など、検討した内容を書く。
**新たに検討する必要がない場合も、その理由を一文書く**(「この設計は外部入力・機微データを扱わないため新たな検討は不要」など)。理由を書くことで、レビュアーが見落としを指摘できる。

## 法的考慮事項(不要なら削除)

このプロジェクトで法的検討が要るのは、基本的に外部 OSS/ライブラリを導入するときのライセンス検討くらいである。
それ以外で法令違反になる可能性は薄いので、**法的検討が不要なら本セクションごと削除してよい**(不要理由も書かなくてよい)。

## 負荷・コスト

この設計が **ランタイム・運用で背負うコスト**(submit ごと・frame ごとの CPU / GPU 仕事、メモリ、アロケーション、データ量に対するスケール挙動など)と、それが設計の妥当性に与えるトレードオフを書く。
**実装の工数・実装範囲(何ファイル・何関数を変えるか、難易度、スケジュール)は書かない**(README「『負荷・コスト』はランタイムの負荷であって実装工数ではない」)。
考慮する必要がない場合は、なぜ考えなくてよいのか(例:「hot path を増やさない」)を一文書く。

## 未解決の論点(省略可)

「この論点は現時点では決めず、意図して先送りする」という設計判断を、その妥当性の根拠とともに書く(先送り自体が設計判断であり、他の節と同じ水準の根拠を要する)。
書くべき内容は次の 4 点である。

- なぜ今決めないことが妥当か(例: 現時点では判断材料が足りず、先まで実装しないと分からない。決め切るために時間を使うより、先にミニマルな価値を届けるほうがよい)。
- 判断を先送りしたときの影響。
- 先送りして後で問題が発覚した場合に、どの程度の手戻り・問題になりうるかの見積もり。
- 決めるための次の一手(いつ・何が揃ったら決めるか)。

議論がまとまらない・解決案が思いつかないという消極的な理由で論点をここへ移して執筆やレビューを終えてはならない(それは doc が未完成なのであって、未解決の論点ではない)。
approve 時点でこの節に残ってよいのは、上記の根拠を伴う積極的な先送りだけである。
