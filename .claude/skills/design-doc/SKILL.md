design doc を **執筆 → 多観点の敵対的レビュー → 再決定+再執筆** のラウンドで仕上げ、blocker/major が出なくなった `ready for review` の完成品をユーザーに提示する。
敵対的レビューは AI による品質ゲートであり、人間レビュー(レビュー PR)を置き換えない。
最終承認(`approved` / `rejected` の確定)は必ず人間が行う。
dynamic workflow の実行は、このスキルが起動された時点で明示的にオプトインされているものとして扱う。

このスキルは収束ラウンドの **orchestrator の手順** を定める。
執筆の workflow(設計競作スクリプト A・執筆競作+統合スクリプト B・骨子の形式)は `.claude/skills/design-doc-write/SKILL.md`、レビューの workflow(観点ファンアウトのスクリプト)は `.claude/skills/design-doc-review/SKILL.md` にあり、必要な時点で Read して使う。

## 全体フロー

1. **執筆前のすり合わせ。** 記録する意思決定を1つに定め、目的・解くべき問題・スコープの外枠をユーザーとすり合わせる(曖昧なら `grilling` で1問ずつ)。書き始めた後は、初稿が完成するまで設計判断・分割・スコープをユーザーに仰がない(正: docs/design/README.md「初稿は執筆側が完成させる」)。
2. **初稿の執筆。** `design-doc-write` の手順に従う(設計競作 → 骨子確定 → 執筆競作+統合)。骨子ファイルは収束ラウンドでも使うので残す。
3. **収束ラウンド(下記)。** blocker/major が出なくなるまで回す。
4. **仕上げ。** status を `ready for review` にし、INDEX.md を更新し、`feature/design-<topic> → develop` を `--no-ff`(PR なし)で landing する(docs/harness/git-and-pr.md「Design Doc のブランチ運用」)。各ラウンドの指摘と対応を要約し、doc の場所を伝えてユーザーにレビュー開始を促す。
5. **レビュー PR。** ユーザーがレビュー開始を宣言したら `/pr-workflow`「Design Doc レビュー PR」に従う(以降のマージはユーザー)。

既存 doc のレビュー+修正から始める場合は、まず対象 doc から骨子(決定・代替案・セクション別の主張)を書き起こしてから、収束ラウンドに入る。

## 収束ラウンド(orchestrator が回す)

各ラウンドは「レビュー(workflow)→ orchestrator が裁く → 再執筆(workflow)」で、**ラウンド間の判断はスクリプトに自動化させず orchestrator(あなた)が行う**。

1. **レビュー。** `design-doc-review` のスクリプトを実行する。関連する draft(相互参照する/同じ設計テーマを分担する)は `docPaths` にセットで列挙する。戻り値の `complete` が false(レビュアーの過半が走っていない)なら、そのラウンドを収束判定に使わず回し直す。
2. **裁く。** blocker/major を自分で読み、処理を決める。
   - **kind=design(決定の反転・構造・スコープ/分割の変更を要する)**: その論点を哲学(docs/philosophy.md)・各案の Pros/Cons・コードベースの一貫性で **自分が再決定**し、骨子を更新する(決定を差し替えるか、指摘を退けて理由を代替案/落とし穴の主張に反映する)。再執筆エージェントに決定の反転を委ねない。doc の分割・統合が要るなら、骨子とファイル構成も自分で組み替える。
   - **kind=writing**: findings としてそのまま再執筆に渡す。
   - **reject 推奨**: 自分も設計が根本的に立たないと判断したら、reject 推奨とその理由を決めた立場として doc に記録して初稿を仕上げる。ユーザーに「どうしますか」と問わない(最終 approve / reject の確定だけがユーザーの仕事)。
3. **再執筆。** 更新した骨子と findings で `design-doc-write` のスクリプト B(執筆競作+統合)を実行する。**blocker/major を単一エージェントの付け足し編集で直さない**(構造的な問題は骨子から書き直してこそ収束する。付け足しはレビュー対象面を増やして doc を肥大・発散させる)。直接編集してよいのは minor/nit だけが残ったときの、構造を変えない整形に限る。
4. **終了判定とガード。**
   - blocker/major がゼロになったら、minor を軽量整形して終了。
   - 最大 5 ラウンド。blocker/major が2ラウンド連続で減らなければ打ち切る(機械的な再執筆では詰め切れない合図)。
   - **打ち切りはユーザーに問う合図ではない。** 敵対的レビュアーは十分長い doc から常に何かを見つけるので、機械的に 0 収束しないのは正常。残る指摘は kind=design の手順(自分で再決定)で決め切り、初稿を完成させる(正: README「初稿は執筆側が完成させる」)。
   - **発散の兆候**(ラウンドをまたいで doc が膨らみ続ける・骨子に無いセクションや改名が現れる・指摘が減らない)を見たら、肥大版を磨き続けず、最後に landing した版に戻して骨子の設計方針を直してから書き直す。復元は `git restore --source=HEAD --staged --worktree -- <paths>` のように対象を名指しする(`git reset --hard` は自動許可されないことがある)。
   - スクリプト B の戻り値 `incomplete` は docs 未更新を意味する。そのラウンドを完了扱いにせず、原因(サーバ過負荷・レート制限)が解けてから回し直す。

## 注意

- **精度優先。** コストを理由にラウンド数・執筆候補数・レビュー観点を勝手に削らない。
- **持続並列は 6 まで。ワークフローは1本ずつ。** 2本以上の同時起動はサーバ側スロットルの HTTP 429 を誘発する(短時間エージェントの瞬間的な試行では 429 が出ないため「もっと並列できる」と誤判断しない)。
- **Workflow の `args` に頼らない。** 対象 doc・骨子パスはスクリプト定数を直接書き換え、実行結果の `docPaths` が意図と一致するか確認する。
- **セットは骨子で整合を取り、執筆・再執筆は doc ごとに回す**(スクリプト B が per-doc で回す)。セット全体を1エージェントに一括で書かせない。
- ハーネス(このスキル等)の更新自体は Issue と PR で管理する(CLAUDE.md)。
