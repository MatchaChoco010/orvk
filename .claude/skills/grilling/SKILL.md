対象を問わず（計画・設計・要件・前提・問題設定など）、共通理解に達するまでユーザーに1問ずつ徹底的に確認する。
論点の分岐を一つずつ辿り、前提・抜け・依存関係を順に解消して穴を突く。

## 進め方

- **1問ずつ。** 前の回答を受けてから次の質問を組み立てる（前の回答が後続の選択肢を変える）。複数の質問を一度に投げない。
- **コードで答えが出ることは訊かない。** 質問を組む前に、その論点がコード調査で確定できないか確認し、できるなら自分で調べる。
- **前提を説明してから選択肢を出す。** 質問に専門用語・内部型名・独自概念を使うなら、その語が何を指すか・現状・各案の意味を先に説明する。ユーザーは背景（未完成の資料・コードの詳細等）を共有していない前提で、質問だけで判断できるようにする。説明なしにいきなり選択肢を出さない。
- **推奨案を必ず添える。** 最も推奨する案を先頭に置き、なぜ推奨かを述べる。

## 質問は番号付きチャットで出す

各質問は通常のチャットに番号付きの選択肢を書き、ユーザーに番号で答えてもらう（選択肢の個数制限なし）。
末尾に「自由記述回答」用と「逆質問」用の番号を1つずつ置き、回答の書き方の例を質問文に必ず添える。
形は次のとおり。

```
（前提説明：用語・現状・各案の意味を、背景を共有していない人にも分かるように書く）

1. 案A（推奨） — 含意・トレードオフ。推奨理由。
2. 案B — …
3. 案C — …
4. ✍️ 選択肢にない回答を自由記述する
5. 🔁 逆質問する（この質問にはまだ答えない）

番号で答えてください。書き方の例：
- `2` … 案B を選ぶ
- `4: 実際の回答をここに書く` … 選択肢にない回答を自由記述する
- `5: 聞きたいことをここに書く` … 逆質問する
```

返信を番号で判別する：通常の選択肢番号 → その決定。
自由記述番号 + 記述 → 選択肢にない回答。
逆質問番号 + 記述 → 逆質問。

## 逆質問の扱い

逆質問（逆質問番号で来た問い）は回答ではない。
誠実に答えてから問い直し、実際の決定（選択肢番号 or 自由記述回答）が来て初めて次へ進む。
逆質問やそれへの回答を決定と解釈して先へ進まない。
完全に同一の質問を出し直す必要はない：逆質問・指摘で前提が変わる/勘違いが判明したら、質問・選択肢を更新して問い直してよい。

進め方・逆質問の扱いはエージェントへの内部指示であり、裏で守る。
手順の復唱・進行アナウンスをユーザー向けチャットに書かない。
